# Secrets and environment files

Dockyard does not include a secret manager. It keeps Docker Compose as the runtime and gives operators safer ways to prepare values and environment files.

## Recommended pattern

Use this split:

```text
team-dashboard/
  Dockyard.yaml
  values.yaml
  values.schema.json
  compose.yaml

deploy-values/
  dashboard-prod.yaml
  dashboard-prod.env.example
```

The package author owns the reusable package. The operator owns the deployment values.

## Use environment references for secrets

In a private deployment values file, use an environment-variable reference instead of the real secret:

```yaml
database:
  user: dashboard
  password: "${DASHBOARD_DB_PASSWORD}"
```

Then the Compose template can use the Dockyard value normally:

```yaml
services:
  db:
    environment:
      POSTGRES_PASSWORD: "${database.password}"
```

Dockyard renders this to:

```yaml
services:
  db:
    environment:
      POSTGRES_PASSWORD: "${DASHBOARD_DB_PASSWORD}"
```

Docker Compose then resolves `DASHBOARD_DB_PASSWORD` from the environment of the Dockyard process.

## PowerShell example

```powershell
$env:DASHBOARD_DB_PASSWORD = "use-a-long-random-password"

.\bin\dockyard.exe install dashboard-prod .\team-dashboard `
  -f .\deploy-values\dashboard-prod.yaml
```

## Bash example

```bash
export DASHBOARD_DB_PASSWORD='use-a-long-random-password'

dockyard install dashboard-prod ./team-dashboard \
  -f ./deploy-values/dashboard-prod.yaml
```

## Generate a `.env.example`

Generate variables from all values:

```bash
dockyard env template ./team-dashboard \
  -o ./deploy-values/dashboard-prod.env.example
```

Generate only secret-looking variables:

```bash
dockyard env template ./team-dashboard \
  --sensitive-only \
  -o ./deploy-values/dashboard-prod.secrets.env.example
```

Add a prefix:

```bash
dockyard env template ./team-dashboard \
  --prefix dashboard \
  -o ./deploy-values/dashboard.env.example
```

Example output:

```dotenv
# Generated by Dockyard.
# Keep real production .env files private and out of source control.
# Prefer CI/CD secrets or a dedicated secret manager for production.

DATABASE_PASSWORD=
```

## Check an env file

```bash
dockyard env check ./deploy-values/dashboard-prod.env.example
```

The check detects:

```text
duplicate keys
malformed lines
populated secret-like keys
```

Use strict mode in CI:

```bash
dockyard env check ./deploy-values/dashboard-prod.env.example --strict
```

## What to commit

Good candidates for Git:

```text
values.yaml
values.schema.json
deploy-values/*.example.yaml
deploy-values/*.env.example
```

Usually do not commit:

```text
.env
*.env
deploy-values/prod.yaml with real secrets
private keys
certificates
tokens
```

## Notes

`dockyard env check` warns when secret-like keys have populated values. That is intentional. A committed `.env.example` should usually contain blank placeholders for secret-like values.
