# Security

Dockyard adds security checks and safer workflows around Docker Compose, but it does not replace a secret manager or container hardening review.

## Policy commands

List built-in checks:

```bash
dockyard policy list
```

Run policy checks against a package, archive, or OCI package:

```bash
dockyard policy check ../dockyard-work/team-dashboard \
  -f ../deploy-values/dashboard-prod.yaml
```

Scan package defaults for populated secret-like values:

```bash
dockyard secrets scan ../dockyard-work/team-dashboard --strict
```

## Policy checks

`Dockyard.yaml` can enable checks such as:

```yaml
security:
  requireNonRoot: true
  requireHealthchecks: true
  requireReadOnlyRootFilesystem: false
  requireNoNewPrivileges: true
  requireCapDropAll: true
  disallowPrivileged: true
  disallowHostNetwork: true
  disallowDockerSocketMount: true
  disallowHostPathMounts: false
  disallowLatestTag: true
```

Dockyard focuses on common high-impact risks:

```text
privileged containers
host networking
Docker socket mounts
host-path mounts
root containers
missing health checks
missing no-new-privileges
missing cap_drop: ALL
missing read_only
latest or untagged images
```

## Secret handling

Dockyard does not try to be a secret manager.

Recommended model:

```text
Package author owns:
  Dockyard.yaml
  compose.yaml
  values.yaml
  values.schema.json

Operator owns:
  ../deploy-values/prod.yaml
  ../deploy-values/prod.env
```

Keep real production secrets out of reusable packages and package archives.

Use environment references in private values files:

```yaml
database:
  password: "${DASHBOARD_DB_PASSWORD}"
```

Then provide the value through a shell, CI/CD secret store, or private env file:

```bash
dockyard install dashboard-prod ../dockyard-work/team-dashboard \
  -f ../deploy-values/dashboard-prod.yaml \
  --env-file ../deploy-values/dashboard-prod.env
```

## Advanced hardening

The package generated by `dockyard init` is intentionally runnable with the stock `nginx` image. It keeps useful defaults such as `no-new-privileges`, but does not enable every hardening option by default.

Some container images need writable cache, temp, or runtime directories. For example, stock `nginx` may fail with `read_only: true` unless writable paths are provided.

A hardened service may look like this:

```yaml
services:
  web:
    image: "${image.repository}:${image.tag}"
    user: "101:101"
    read_only: true
    tmpfs:
      - /var/cache/nginx
      - /var/run
      - /tmp
    security_opt:
      - no-new-privileges:true
    cap_drop:
      - ALL
```

Use the stronger checks only when the image is known to support them, or when the package provides required writable paths.
