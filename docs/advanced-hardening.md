# Advanced hardening

The package generated by `dockyard init` is intentionally runnable with the stock `nginx` image. It keeps useful defaults such as `no-new-privileges`, but it does not enable every hardening option by default.

Some container images need writable cache, temp, or runtime directories. For example, stock `nginx` may fail with `read_only: true` unless writable paths are provided.

## Runnable starter

```yaml
services:
  web:
    image: "${image.repository}:${image.tag}"
    ports:
      - "${service.port}:80"
    healthcheck:
      test: ["CMD", "nginx", "-t"]
      interval: 30s
      timeout: 5s
      retries: 3
    security_opt:
      - no-new-privileges:true
```

## More hardened nginx example

This example shows the general pattern for moving toward a read-only root filesystem while still providing writable runtime paths. Test image-specific behavior before using it in production.

```yaml
services:
  web:
    image: "${image.repository}:${image.tag}"
    ports:
      - "${service.port}:80"
    read_only: true
    tmpfs:
      - /var/cache/nginx
      - /var/run
      - /tmp
    healthcheck:
      test: ["CMD", "nginx", "-t"]
      interval: 30s
      timeout: 5s
      retries: 3
    security_opt:
      - no-new-privileges:true
```

For stricter packages, enable matching policy checks in `Dockyard.yaml`:

```yaml
security:
  requireReadOnlyRootFilesystem: true
  requireNoNewPrivileges: true
```

Only enable `requireCapDropAll` or `requireNonRoot` after confirming the target image supports those constraints.
