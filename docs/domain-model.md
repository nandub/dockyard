# Domain Model

This document defines Dockyard domain concepts observed in source and documentation.

## Core Concepts

- Dockyard package: directory or `.dockyard.tgz` archive containing a package manifest and related files.
- Package manifest: `Dockyard.yaml`.
- Values: defaults and overrides used to render Compose templates.
- Values schema: `values.schema.json`, used for JSON Schema validation.
- Rendered Compose: Docker Compose YAML generated by Dockyard.
- Release: named deployment tracked in Dockyard state.
- Revision: persisted release snapshot under a release.
- Lockfile: `dockyard.lock`, generated and verified by `internal/lock`.
- Provenance metadata: `package.provenance.json`, generated package metadata. No signature verification was observed.
- Catalog: metadata index used for `catalog://NAME[:VERSION]` and shorthand resolution.
- OCI package: Dockyard package archive pushed to or pulled from an OCI registry using ORAS.

## Runtime Boundary

Dockyard owns package loading, values merging, rendering, validation, policy checks, packaging, catalog resolution, OCI orchestration, and local release state.

Docker Compose owns final Compose interpretation, image pulls, networks, volumes, containers, and runtime status.

## Extension Boundary

No first-class plugin loader was observed. Dockyard extension points are data/configuration based: packages, Compose files, values, schemas, overlays, dependency metadata, catalog entries, and OCI references.

Historical evidence: `.ai/onboarding/reports/plugin-model.md`.
