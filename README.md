# Dockyard

Dockyard is a package manager, renderer, release tracker, and security linter for Docker Compose applications.

It renders Dockyard packages to plain Docker Compose YAML, records local release revisions, and delegates runtime operations to `docker compose`.

## Compatibility notes

Dockyard delegates runtime behavior to Docker Compose. The starter package generated by `dockyard init` is intentionally runnable with stock `nginx`; stricter hardening is documented separately because some images need writable runtime directories.

## v0.8 MVP commands

Beginner path without Compose overlays:

```bash
dockyard init ./example-app
dockyard values template ./example-app -o deploy-values/prod.yaml
dockyard values validate ./example-app -f deploy-values/prod.yaml
dockyard env template ./example-app -o deploy-values/example-app.env.example
dockyard env check deploy-values/example-app.env.example
dockyard lint ./example-app -f deploy-values/prod.yaml
dockyard render ./example-app -f deploy-values/prod.yaml --explain
dockyard config ./example-app -f deploy-values/prod.yaml
dockyard secrets scan ./example-app
dockyard policy check ./example-app -f deploy-values/prod.yaml
dockyard lock ./example-app -f deploy-values/prod.yaml
dockyard package ./example-app --locked -f deploy-values/prod.yaml -o example-app-0.1.0.dockyard.tgz
dockyard verify example-app-0.1.0.dockyard.tgz -f deploy-values/prod.yaml --require-lock
dockyard install myapp example-app-0.1.0.dockyard.tgz -f deploy-values/prod.yaml --require-lock
dockyard status myapp --json
dockyard inspect myapp
dockyard list
dockyard uninstall myapp --dry-run
dockyard doctor
```

Advanced path with a structural Compose overlay:

```bash
dockyard lock ./example-app -f deploy-values/prod.yaml --overlay prod
dockyard package ./example-app --locked -f deploy-values/prod.yaml --overlay prod -o example-app-0.1.0.dockyard.tgz
dockyard install myapp example-app-0.1.0.dockyard.tgz -f deploy-values/prod.yaml --overlay prod --require-lock
```

Use values files for environment-specific settings. Use overlays only when the Compose structure itself needs to change.



## Secrets and environment ergonomics

Dockyard supports a simple production-safe pattern:

1. Keep reusable package defaults in `values.yaml`.
2. Keep environment-specific overrides in `deploy-values/*.yaml`.
3. For secrets, put environment-variable references in the private values file, for example `${DASHBOARD_DB_PASSWORD}`.
4. Provide the real secret through your shell, CI/CD secret store, or a private `.env` file before running Dockyard.

Generate an operator-friendly `.env` template:

```bash
dockyard env template ./team-dashboard -o deploy-values/team-dashboard.env.example
```

Generate only secret-like variables:

```bash
dockyard env template ./team-dashboard --sensitive-only -o deploy-values/team-dashboard.secrets.env.example
```

Validate a `.env` file for duplicate keys, malformed lines, and populated secret-like keys:

```bash
dockyard env check deploy-values/team-dashboard.env.example
```

`dockyard env check` intentionally warns when a secret-like key has a populated value. Commit `.env.example` files, not production `.env` files.

## Security policy

List built-in policy checks:

```bash
dockyard policy list
```

Run policy checks against a package, archive, or OCI package:

```bash
dockyard policy check ./team-dashboard -f ./deploy-values/dashboard-prod.yaml
```

Scan package defaults for populated secret-like values:

```bash
dockyard secrets scan ./team-dashboard --strict
```

Dockyard intentionally does not store or manage production secrets. Keep operator values files private and prefer external secret managers for production deployments.

## Compose validation

Validate the final rendered Compose file without installing anything:

```bash
dockyard config ./team-dashboard -f ./deploy-values/dashboard-prod.yaml
```

Validate while rendering:

```bash
dockyard render ./team-dashboard \
  -f ./deploy-values/dashboard-prod.yaml \
  --validate-compose
```

Dockyard renders standard Compose YAML, but Docker Compose remains the source of truth for runtime behavior. See [Docker Compose compatibility](docs/compose-compatibility.md).

## Values templates

Dockyard can generate an operator-friendly values override file from `values.yaml` and `values.schema.json`:

```bash
dockyard values template ./team-dashboard -o ./deploy-values/dashboard-prod.yaml
```

The generated file preserves safe defaults, adds comments from schema `description` fields, and masks sensitive fields such as passwords, secrets, tokens, credentials, and fields marked with `x-dockyard-sensitive: true`.

Validate an override file before rendering or installing:

```bash
dockyard values validate ./team-dashboard -f ./deploy-values/dashboard-prod.yaml
```

Print the package schema:

```bash
dockyard values schema ./team-dashboard
```

Example schema field with Dockyard metadata:

```json
{
  "type": "string",
  "description": "Password for the PostgreSQL application user.",
  "x-dockyard-sensitive": true
}
```


## OCI registry support

Dockyard v0.6.2 can push and pull `.dockyard.tgz` package archives using OCI registries through the `oras` CLI.

Install and authenticate with `oras` before using these commands. Dockyard does not log registry credentials; authentication is delegated to the local `oras` configuration.

Push a verified archive:

```bash
dockyard push team-dashboard-0.1.0.dockyard.tgz \
  oci://ghcr.io/nandub/dockyard/team-dashboard:0.1.0
```

Pull an archive:

```bash
dockyard pull oci://ghcr.io/nandub/dockyard/team-dashboard:0.1.0 \
  -o team-dashboard-0.1.0.dockyard.tgz
```

Install directly from an OCI reference:

```bash
dockyard install dashboard-prod \
  oci://ghcr.io/nandub/dockyard/team-dashboard:0.1.0 \
  -f ./deploy-values/dashboard-prod.yaml \
  --require-lock
```

Upgrade or diff directly from OCI:

```bash
dockyard diff dashboard-prod \
  oci://ghcr.io/nandub/dockyard/team-dashboard:0.1.1 \
  -f ./deploy-values/dashboard-prod.yaml \
  --require-lock

dockyard upgrade dashboard-prod \
  oci://ghcr.io/nandub/dockyard/team-dashboard:0.1.1 \
  -f ./deploy-values/dashboard-prod.yaml \
  --require-lock
```

`dockyard doctor` reports whether `oras` is available. OCI references must include an explicit tag or digest.

See `docs/oci.md` for OCI registry details.

## State location

Dockyard stores release state in this precedence order:

1. `--home`
2. `DOCKYARD_HOME`
3. `~/.dockyard`

Example:

```bash
export DOCKYARD_HOME=/var/lib/dockyard
dockyard install myapp ./example-app
```

Release layout:

```text
~/.dockyard/
  releases/
    myapp/
      current
      revisions/
        1/
          Dockyard.yaml
          values.yaml
          dockyard.lock
          compose.rendered.yaml
          release.json
```

## Package format

A package directory contains:

```text
Dockyard.yaml
values.yaml
values.schema.json
compose.yaml
README.md
SECURITY.md
```

`dockyard package` creates a `.dockyard.tgz` archive and includes:

```text
SHA256SUMS
package.provenance.json
dockyard.lock        # when generated before packaging
```

Package archives reject common secret-like files such as `.env`, `*.pem`, `*.key`, `id_rsa`, and `id_ed25519`.

## Lockfiles

`dockyard lock` creates `dockyard.lock` for a specific render:

```bash
dockyard lock ./team-dashboard \
  -f ./deploy-values/dashboard-prod.yaml
```

The lockfile records:

- package identity
- values digest
- rendered Compose digest
- package file digests
- images found in rendered Compose
- digest-pinned image references when already present

Lockfiles are intentionally local and deterministic. Dockyard v0.6.1 does **not** contact image registries to resolve image tags to digests.

Use `--require-lock` during install or upgrade to ensure the lockfile still matches the package, values, and rendered Compose output:

```bash
dockyard install dashboard-prod team-dashboard-0.1.0.dockyard.tgz -f ./deploy-values/dashboard-prod.yaml --require-lock
dockyard upgrade dashboard-prod ./team-dashboard -f ./deploy-values/dashboard-prod.yaml \
  --require-lock
```

Use `--locked` when packaging to require `dockyard.lock` and record lock provenance:

```bash
dockyard package ./team-dashboard --locked -f ./deploy-values/dashboard-prod.yaml -o team-dashboard-0.1.0.dockyard.tgz
```

## Installing from archives

v0.6.1 can install either a package directory or a local archive:

```bash
dockyard install dashboard-prod ./team-dashboard
dockyard install dashboard-prod team-dashboard-0.1.0.dockyard.tgz \
  -f ./deploy-values/dashboard-prod.yaml \
  --require-lock
```

Archives are verified before extraction for install, upgrade, and diff.

## Security defaults

Dockyard rejects or warns on risky Compose patterns when configured in `Dockyard.yaml`:

- privileged containers
- host networking
- Docker socket mounts
- root containers
- missing health checks
- `latest` or untagged images

The repository includes CI plus a security workflow for `govulncheck`, `staticcheck`, and Semgrep.
